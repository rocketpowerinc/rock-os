# Logging In Linux

Logging answers a different question than monitoring:

```text
What happened?
```

Monitoring is the live dashboard. Logging is the memory. When a service fails,
a system reboots, a login is denied, a disk complains, or an update breaks
something, logs are where Linux writes the receipts.

Logs can be boring. Logs can be noisy. Logs can also save you six hours of
guessing, which is why we learn them.

## Why Logs Matter

Logs help you:

- Troubleshoot failed services
- Investigate crashes
- Review login attempts
- Track package changes
- Find hardware errors
- Understand boot problems
- Detect suspicious activity
- Prove what happened instead of arguing with vibes

If monitoring says "something is wrong," logging often tells you what.

## The Two Big Logging Worlds

On many modern Linux systems, logs live in two main places:

```text
systemd journal
/var/log/
```

The systemd journal is queried with:

```bash
journalctl
```

Traditional log files live under:

```bash
/var/log/
```

Some distros lean heavily on the journal. Others still write many classic log
files. Most use both in some way, because Linux enjoys giving you options and
then pretending that is the same thing as simplicity.

## journalctl Basics

To see recent logs:

```bash
journalctl
```

This can be a lot. A lot-lot. The kind of lot that makes your terminal feel
like it has opened a firehose.

Show logs from the current boot:

```bash
journalctl -b
```

Show warnings and worse from the current boot:

```bash
journalctl -p warning -b
```

Follow logs live:

```bash
journalctl -f
```

Follow logs for a specific service:

```bash
journalctl -u ssh -f
```

Check logs for a service from the current boot:

```bash
journalctl -u NetworkManager -b
```

This is one of the most useful troubleshooting commands on a systemd distro.

## Log Priority Levels

The journal supports priority levels.

Common levels:

| Priority | Meaning |
| --- | --- |
| `emerg` | System is unusable |
| `alert` | Immediate action needed |
| `crit` | Critical condition |
| `err` | Error |
| `warning` | Warning |
| `notice` | Normal but notable |
| `info` | Informational |
| `debug` | Debug details |

Examples:

```bash
journalctl -p err -b
journalctl -p warning -b
journalctl -p info -u NetworkManager
```

If you are troubleshooting, start with warnings and errors. Debug logs are
useful, but they can also bury you alive in detail.

## Logs From Previous Boots

List boots:

```bash
journalctl --list-boots
```

You might see:

```text
-2  7c2b...  Fri 2026-05-15 09:12:01
-1  84af...  Sat 2026-05-16 14:44:19
 0  a91d...  Sun 2026-05-17 08:22:03
```

Show logs from the previous boot:

```bash
journalctl -b -1
```

Show errors from the previous boot:

```bash
journalctl -p err -b -1
```

This is extremely useful after a crash. The current boot may look clean because
the crime already happened. You need the previous boot's witness statement.

## Kernel Logs

Kernel messages are useful for hardware, drivers, disks, USB devices, graphics,
network adapters, and boot problems.

Show kernel logs from the current boot:

```bash
journalctl -k -b
```

Or:

```bash
dmesg
```

For human-readable timestamps:

```bash
dmesg -T
```

Look here for:

- Disk errors
- USB issues
- Driver failures
- Firmware warnings
- GPU problems
- Filesystem errors
- Hardware detection messages

If Linux says your hardware did something weird, the kernel log is often where
it mutters about it.

## Traditional Logs In /var/log

List logs:

```bash
ls -lh /var/log
```

Common files and folders:

| Path | Purpose |
| --- | --- |
| `/var/log/syslog` | General system logs on Debian/Ubuntu-style systems |
| `/var/log/auth.log` | Authentication logs on Debian/Ubuntu-style systems |
| `/var/log/messages` | General logs on some Fedora/RHEL-style systems |
| `/var/log/secure` | Authentication logs on some Fedora/RHEL-style systems |
| `/var/log/apt/` | APT package history |
| `/var/log/pacman.log` | Arch package history |
| `/var/log/dnf.log` | Fedora DNF logs |
| `/var/log/Xorg.0.log` | Xorg display server logs, when used |

Not every distro has every file. If a file is missing, that does not always
mean something is broken. It may just mean your distro logs that information
somewhere else.

## Reading Log Files

Use `less` for reading:

```bash
less /var/log/syslog
```

Follow a log live:

```bash
tail -f /var/log/syslog
```

Show the last 100 lines:

```bash
tail -n 100 /var/log/syslog
```

Search inside a log:

```bash
grep -i "error" /var/log/syslog
```

Search recursively:

```bash
grep -Ril "failed" /var/log
```

Some logs require root:

```bash
sudo less /var/log/auth.log
```

## Authentication Logs

Authentication logs show logins, sudo usage, SSH attempts, failed passwords,
and other identity-related events.

On Debian/Ubuntu:

```bash
sudo less /var/log/auth.log
```

On Fedora/RHEL-style systems:

```bash
sudo less /var/log/secure
```

With systemd:

```bash
journalctl _COMM=sshd
journalctl -u ssh
journalctl -u sshd
```

Look for:

- Failed login attempts
- Successful SSH logins
- Repeated sudo failures
- Unknown usernames
- Logins at strange times

One failed login is not always a crisis. Five thousand failed SSH attempts from
the internet is the server equivalent of someone rattling the doorknob all
night.

## Package Logs

Package logs tell you what changed on the system.

Debian/Ubuntu:

```bash
less /var/log/apt/history.log
less /var/log/dpkg.log
```

Arch:

```bash
less /var/log/pacman.log
```

Fedora:

```bash
less /var/log/dnf.log
```

These are useful when something worked yesterday and broke today. Check what
was installed, removed, or upgraded.

## Log Rotation

Logs cannot grow forever. If they did, your disk would eventually become a
museum of old complaints.

Traditional log files are usually managed by `logrotate`.

Check its config:

```bash
ls /etc/logrotate.d
cat /etc/logrotate.conf
```

The systemd journal has its own size controls.

Check journal disk usage:

```bash
journalctl --disk-usage
```

Clean old journal logs by time:

```bash
sudo journalctl --vacuum-time=14d
```

Clean by size:

```bash
sudo journalctl --vacuum-size=1G
```

Do not delete logs blindly if you are investigating a problem. Logs are
evidence. Clean them after you understand what happened.

## Persistent Journal Logs

Some systems keep journal logs only in memory by default, meaning logs from
previous boots may not be available.

To enable persistent journal logs:

```bash
sudo mkdir -p /var/log/journal
sudo systemctl restart systemd-journald
```

Then check:

```bash
journalctl --list-boots
```

Persistent logs are useful for debugging crashes, failed boots, and random
shutdowns. If the system forgets everything after reboot, it is hard to ask it
what went wrong.

## Security Logging Habits

For a personal Linux machine, build a few simple habits:

- Check failed services after boot.
- Review warnings after updates.
- Watch authentication logs if SSH is enabled.
- Know which logs your distro uses.
- Keep enough log history to troubleshoot.
- Do not expose SSH to the internet casually.
- If something feels wrong, save relevant logs before rebooting repeatedly.

Useful quick checks:

```bash
systemctl --failed
journalctl -p warning -b
journalctl -p err -b
sudo ss -tulpn
```

For SSH:

```bash
journalctl -u ssh -b
journalctl -u sshd -b
```

The service name may be `ssh` or `sshd` depending on the distro.

## A Simple Troubleshooting Flow

When something breaks:

1. Check failed services.
2. Check warnings and errors from this boot.
3. Check the specific service logs.
4. Check kernel logs.
5. Check package logs if it broke after updates.
6. Check previous boot logs if the machine crashed.
7. Save useful output before making big changes.

Example:

```bash
systemctl --failed
journalctl -p warning -b
journalctl -u NetworkManager -b
journalctl -k -b
```

## Final Thoughts

Logs are not glamorous, but they are honest. Mostly.

They will not always tell you the answer directly, but they will usually tell
you where to look next. Learn `journalctl`, learn `/var/log`, and learn how to
compare what changed.

When Linux breaks, do not just guess. Ask the logs. They were there.
