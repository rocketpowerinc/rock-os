# Linux Servers

Not every Linux machine needs a desktop.

In fact, many of the most important Linux machines you will ever touch will not
have a graphical interface at all. No wallpaper. No dock. No app menu. No
animated settings panel politely asking if you want dark mode.

Just a shell, services, logs, networking, storage, and responsibility.

That is a Linux server.

## What Is A Server?

A server is a computer that provides services to other computers, users, or
devices.

Examples:

- File server
- Web server
- Database server
- Backup server
- Media server
- DNS server
- VPN server
- Git server
- Home automation server
- Container host
- Virtual machine host

A server can be a rack-mounted machine in a datacenter, a small mini PC in your
closet, a Raspberry Pi, an old laptop, or a virtual machine running on another
computer.

The job matters more than the shape.

## Why No Desktop?

Servers usually do not need a desktop environment because the machine is not
there to be used like a normal workstation. It is there to run services.

No desktop means:

- Fewer packages
- Less memory usage
- Fewer background processes
- Smaller attack surface
- Fewer updates
- Less graphical complexity
- Easier remote management

If a server's job is to run Docker containers, host files, serve a website, or
manage backups, a full desktop is usually unnecessary. Installing a desktop on
a server can be like putting a gaming chair in an electrical closet. Stylish,
maybe, but not solving the real problem.

## Headless Linux

A server without a monitor, keyboard, or mouse is called headless.

Headless systems are managed remotely, usually over SSH. You boot the machine,
it joins the network, and you connect from another computer.

The server might sit quietly in a corner for months. If configured well, that is
the dream. A good server is boring in the same way a good roof is boring: you
notice it most when it fails.

## SSH Is The Main Door

SSH is the main way to manage Linux servers.

Connect to a server:

```bash
ssh user@192.168.1.50
```

Connect with a hostname:

```bash
ssh user@server-name
```

Copy a file to a server:

```bash
scp file.txt user@192.168.1.50:/home/user/
```

SSH lets you manage the machine from anywhere on the network. That is powerful,
so secure it properly.

## Basic SSH Hardening

At minimum:

- Use strong passwords or SSH keys.
- Disable root SSH login.
- Keep the server updated.
- Do not expose SSH to the public internet unless you know what you are doing.
- Use a firewall.
- Consider changing the SSH port only as a noise-reduction tactic, not real
  security.

Edit SSH server config:

```bash
sudo nano /etc/ssh/sshd_config
```

Common settings:

```text
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
```

Then restart SSH:

```bash
sudo systemctl restart ssh
```

On some distros the service is named `sshd`:

```bash
sudo systemctl restart sshd
```

> [!WARNING]
> Do not disable password login until you have confirmed SSH key login works.
> Keep one existing SSH session open while testing another. Locking yourself
> out of your own server is a classic rite of passage, but it is not required.

## Server Distros

Most major Linux distros can be servers, but some are especially common.

### Debian

Debian is excellent for servers because it is stable, predictable, and widely
documented. It is my favorite base when I want a server that quietly does its
job without trying to become a lifestyle brand.

### Ubuntu Server

Ubuntu Server is very popular. It has strong hardware support, lots of guides,
and broad cloud/server adoption. The Canonical/company factor still exists, but
for practical server work Ubuntu is common for a reason.

### Fedora Server

Fedora Server is modern and useful if you want newer technologies. It moves
faster than Debian or Ubuntu LTS, so you trade some long-term calm for newer
software.

### Rocky Linux And AlmaLinux

Rocky Linux and AlmaLinux are Red Hat Enterprise Linux-compatible distros. They
are common in enterprise-style environments and useful when you want a
RHEL-like server without paying for RHEL.

These are worth learning if you want to understand business Linux, datacenters,
or environments where stability and long support windows matter.

## Services And Daemons

Servers run services.

A service, sometimes called a daemon, is a background program that provides
some function.

Examples:

- `sshd` for SSH
- `nginx` or `apache2` for web hosting
- `postgresql` or `mariadb` for databases
- `docker` for containers
- `smbd` for Samba file sharing
- `cron` or systemd timers for scheduled jobs

Check a service:

```bash
systemctl status ssh
```

Start a service:

```bash
sudo systemctl start nginx
```

Enable a service at boot:

```bash
sudo systemctl enable nginx
```

Restart a service:

```bash
sudo systemctl restart nginx
```

List failed services:

```bash
systemctl --failed
```

If you understand services, Linux servers become much less mysterious.

## Networking Basics

Servers need predictable networking.

Important concepts:

- IP address
- Hostname
- Gateway
- DNS
- Open ports
- Firewall rules

Check your IP:

```bash
ip addr
```

Check routes:

```bash
ip route
```

Check listening ports:

```bash
sudo ss -tulpn
```

A server should usually have a predictable IP address. You can do that with a
static IP on the server or a DHCP reservation on your router.

For home networks, I usually prefer DHCP reservations on the router. The server
still uses DHCP, but the router always gives it the same address. Clean,
centralized, and less likely to create a tiny networking mystery later.

## Firewalls

A server should only expose the ports it needs.

On Ubuntu/Debian systems, `ufw` is simple:

```bash
sudo ufw allow ssh
sudo ufw enable
sudo ufw status verbose
```

Allow a web server:

```bash
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
```

On Fedora/RHEL-style systems, `firewalld` is common:

```bash
sudo firewall-cmd --add-service=ssh --permanent
sudo firewall-cmd --reload
sudo firewall-cmd --list-all
```

The rule is simple: if a service does not need to be reachable, do not expose
it. A closed door is easier to defend than a hallway full of mystery doors.

## Updates And Reboots

Servers need updates, but they also need uptime.

That means you should update intentionally:

- Know what the server does.
- Read important package changes.
- Update during a low-impact time.
- Reboot when kernel or core system updates require it.
- Check services after reboot.

Debian/Ubuntu:

```bash
sudo apt update
sudo apt upgrade
```

Arch:

```bash
sudo pacman -Syu
```

Fedora:

```bash
sudo dnf upgrade
```

After updates:

```bash
systemctl --failed
journalctl -p warning -b
```

Do not blindly update a critical server five minutes before you need it. That
is not maintenance. That is gambling with a keyboard.

## Logs And Monitoring

Servers should be watched.

Useful quick checks:

```bash
systemctl --failed
journalctl -p warning -b
df -h
free -h
sudo ss -tulpn
```

You want to know:

- Is the disk filling up?
- Are services failing?
- Are there repeated login failures?
- Is the machine running out of memory?
- Are unexpected ports open?
- Are backups succeeding?

Monitoring tells you what is happening now. Logs tell you what happened before
you noticed. Both matter.

## Backups

A server without backups is a temporary server.

Back up:

- Config files
- User data
- Databases
- Container volumes
- Scripts
- SSH keys or access instructions
- Documentation for how the server is built

Important locations:

```text
/etc/
/home/
/srv/
/var/lib/
/opt/
```

The exact paths depend on what the server runs.

Backups should be tested. A backup you have never restored is not a backup. It
is a hopeful rumor.

## When A GUI Makes Sense

Most servers do not need a GUI, but there are exceptions.

A GUI might make sense for:

- Home lab learning
- Media servers
- Machines that are both workstation and server
- Storage appliances with web dashboards
- Beginners who need a bridge while learning

Even then, a web dashboard is often better than a full desktop environment.
Tools like Cockpit can give you browser-based server management without turning
the server into a full graphical workstation.

Install Cockpit on Debian/Ubuntu:

```bash
sudo apt install cockpit
sudo systemctl enable --now cockpit.socket
```

Then visit:

```text
https://server-ip:9090
```

Use dashboards as helpers, not as a replacement for learning the command line.

## A Simple Server Setup Checklist

For a basic Linux server:

1. Install a server-focused distro.
2. Create a normal user.
3. Set a strong password.
4. Configure SSH.
5. Update the system.
6. Set a hostname.
7. Configure predictable networking.
8. Enable a firewall.
9. Install only needed services.
10. Check listening ports.
11. Configure backups.
12. Learn where the logs are.
13. Document what you changed.

That last one matters. Future you will not remember everything. Future you will
pretend they will. Future you is lying.

## Final Thoughts

Linux servers are not magic. They are just Linux machines with a job.

Once you understand SSH, services, networking, firewalls, logs, updates, and
backups, you can run a lot of useful infrastructure on very modest hardware.

Start small. Build one service. Document it. Back it up. Then build the next
one.
